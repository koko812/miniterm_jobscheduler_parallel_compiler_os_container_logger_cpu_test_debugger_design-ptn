# 2周目実装まとめ: step20 手前まで

このメモは、`compiler/re_compiler_book/main.c` を 2 周目として自力で再実装しながら理解した内容を、実装順と設計意図に沿ってまとめたもの。

対象範囲は主に以下。

- 字句解析
- 構文解析
- AST
- ローカル変数
- 制御構文
- 関数呼び出し・関数定義
- `int` 導入
- ポインタ型
- ポインタ経由代入
- ポインタ加算 / 減算
- `sizeof`

## 1. 全体像

現在の実装は、大枠では次の流れになっている。

1. 入力文字列を tokenize する
2. token 列から AST を作る
3. AST に対して型を付与する
4. AST から x86-64 アセンブリを出力する

重要なのは、1周目では曖昧だった以下の責務分離が見えてきたこと。

- `Token`: 入力文字列をどの単語として読むか
- `Node`: 構文木としてどういう式・文か
- `LVar`: 変数名とスタック上の配置
- `Type`: その値や式が何型か
- `Function`: 関数単位のまとまり

短く言うと、

- `Token` = 単語
- `Node` = 構文
- `LVar` = 変数の実体情報
- `Type` = 型情報
- `Function` = 関数単位のコンパイル対象

という分担で考えると理解しやすい。

## 2. 字句解析

### 2.1 基本方針

tokenizer は入力の構造を理解しない。やることは「切る」だけ。

今の tokenizer が主に担当しているのは以下。

- 数字列を `TK_NUM`
- 識別子を `TK_IDENT`
- `return`, `if`, `while`, `for`, `int`, `sizeof` などを予約語トークン
- `+`, `-`, `*`, `&`, `(`, `)` などを記号トークン
- 空白をスキップ
- `==`, `!=`, `<=`, `>=` など複数文字演算子

### 2.2 正規的な処理

ここは正規表現的・有限オートマトン的な世界で処理できる範囲であり、括弧対応や式の入れ子は扱っていない。

たとえば、

- `(` を1個見つけた
- `int` というキーワードを見つけた
- 数字列 `123` を見つけた

という識別だけを行っている。

### 2.3 `char *p` による走査

tokenizer 内では `char *p` を進めて入力を読んでいる。

- `p++` は文字列を1文字先へ進める
- これは C 側のポインタ加算であり、自作言語側のポインタ加算ではない

この点が後で自作言語にポインタ加算を実装するときの実感につながった。

## 3. 構文解析

### 3.1 再帰下降パーサ

構文解析は再帰下降で実装している。

概ね以下の階層になっている。

- `primary`
- `unary`
- `mul`
- `add`
- `relational`
- `equality`
- `assign`
- `expr`
- `stmt`
- `function`
- `program`

演算子優先順位が高いものほど下層にある。

### 3.2 左再帰をループで回避

たとえば加減算は元の文法だと左再帰だが、実装では

- 最初に下位層 (`mul`) を1つ読む
- その後 `+` / `-` が続く限りループで木を左に伸ばす

という形で実装している。

この理解が 2 周目でかなりはっきりした。

### 3.3 `stmt` と `;`

`stmt` は「セミコロンが必要な文」と「不要な文」が混ざるのでややこしい。

最終的に、

- `if` / `while` / `for` / block は即 return
- `int` 宣言文もその場で `expect(";")`
- `return` もその場で `expect(";")`
- 最後の式文だけ `expr(); expect(";")`

という形に整理した。

これにより責務がかなり明確になった。

## 4. AST

### 4.1 `Node`

`Node` は式や文の構造を表す。

主な kind:

- 四則演算: `ND_ADD`, `ND_SUB`, `ND_MUL`, `ND_DIV`
- 比較: `ND_EQU`, `ND_NEQ`, `ND_LTH`, `ND_LEQ`
- 代入: `ND_ASSIGN`
- 数値: `ND_NUM`
- 変数: `ND_LVAR`
- 単項演算: `ND_ADDR`, `ND_DEREF`
- 制御構文: `ND_IF`, `ND_WHILE`, `ND_FOR`, `ND_RETURN`
- 構文単位: `ND_BLOCK`, `ND_FUNCALL`, `ND_NULL`

### 4.2 単項演算

`*` と `&` は二項演算ではなく単項演算なので、`new_node_num(0)` のようなダミーは不要。

最終的には

- `ND_DEREF(lhs=unary(), rhs=NULL)`
- `ND_ADDR(lhs=unary(), rhs=NULL)`

という形にした。

### 4.3 block

`{ ... }` は `ND_BLOCK` として作り、中身は `body` から `next` でつないだ文列にしている。

block は値を返す構文ではないが、現在の codegen では「各文が最終的に stack に1つ値を残す」契約に寄せているため、`ND_BLOCK` の最後に `0` を push する実装にしている。

## 5. ローカル変数

### 5.1 `LVar`

`LVar` は変数そのものの情報。

持っているもの:

- `str`, `len`: 変数名
- `offset`: `rbp` からのオフセット
- `ty`: 変数型
- `next`: locals 用リンク
- `next_param`: params 用リンク

### 5.2 `locals`

`locals` は「現在パース中の関数」の変数表の先頭。

役割:

- `find_LVar()` による名前解決
- offset の割り当て
- 引数と通常ローカルを同じ機構で扱う

### 5.3 暗黙定義の廃止

最初の段階では未定義識別子を見たら勝手に新規 `LVar` を作っていた。

これは

- typo が静かに通る
- 未初期化値を読んで壊れる
- `int` 導入後の意味が薄れる

ので、`int` 導入段階でやめた。

現在は `factor()` で `ident` を読んだら

- `find_LVar()` で見つかれば `ND_LVAR`
- 見つからなければエラー

としている。

## 6. 制御構文

### 6.1 `if`, `while`, `for`

各制御構文は専用ノードを持たせた。

- `ND_IF`: `cond`, `then`, `els`
- `ND_WHILE`: `cond`, `then`
- `ND_FOR`: `init`, `cond`, `inc`, `then`

codegen では `.LbeginN`, `.LelseN`, `.LendN` のようなラベルを発行している。

### 6.2 block 再利用

関数本体の `{ ... }` は `stmt()` 内の `ND_BLOCK` 実装をそのまま再利用している。

これは `function()` 側で別に `{ stmt* }` をもう一度書くより自然だった。

## 7. 関数呼び出しと関数定義

### 7.1 関数呼び出し

`factor()` で

- `ident (...)`

を見たら `ND_FUNCALL` を作る。

引数は `args` を `next` でつないだリストで保持する。

codegen では

1. 各引数式を評価して stack に積む
2. 逆順に `rdi, rsi, rdx, rcx, r8, r9` へ `pop`
3. `call funcname`

としている。

### 7.2 関数定義

関数定義を導入したことで、トップレベルは `stmt*` ではなく `function*` になった。

`Function` 構造体を作った理由は以下。

- 関数名を持ちたい
- 引数リストを持ちたい
- 関数本体を持ちたい
- 関数ごとに locals を持ちたい

`Node` だけに押し込むより責務が明確になった。

### 7.3 `params` と `locals`

ここは 2 周目で理解が深まった箇所。

- `locals`
  - 関数本体中の名前解決用
  - 引数も普通のローカル変数としてここに入れる
- `params`
  - 引数だけを宣言順で保持
  - `rdi, rsi, ...` を `[rbp-offset]` に退避するために使う

重要なのは、

- 同じ `LVar` 実体を共有してよい
- ただしリンクは `next` と `next_param` に分ける

という点。

### 7.4 ABI

関数呼び出しが gcc 由来の外部関数とも噛み合ったのは、System V AMD64 ABI に従っていたから。

- 第1引数: `rdi`
- 第2引数: `rsi`
- 第3引数: `rdx`
- 第4引数: `rcx`
- 第5引数: `r8`
- 第6引数: `r9`

関数定義側ではプロローグでこれらを `[rbp-offset]` に退避している。

## 8. `int` キーワード導入

### 8.1 変数定義

`stmt()` で

- `int x;`

を読めるようにした。

ここでは

1. 型名を読む
2. 識別子を読む
3. `LVar` を作って `locals` に積む
4. `ND_NULL` を返す

という最小構成にしている。

`int x = 3;` はまだやっていない。

### 8.2 関数定義

`function()` は

- `int`
- 関数名
- `(`
- `int ident` の列
- `)`
- block

を読む形に変更した。

## 9. 型システム導入

### 9.1 `Type`

`Type` は以下の形。

- `TY_INT`
- `TY_PTR`
- `ptr_to`

多重ポインタは

- `ptr_to(int)`
- `ptr_to(ptr_to(int))`

のように包んでいくだけで表現できる。

### 9.2 `int_type()` と `ptr_to()`

この2つは「型を読む関数」ではなく「型を作る関数」。

- `int_type()` は `TY_INT`
- `ptr_to(base)` は `TY_PTR` で `ptr_to=base`

`Type *ty; ty->kind = ...` のような未初期化ポインタ書き込みは当然ダメで、`calloc` などで実体を確保してから使う。

### 9.3 `LVar->ty`

宣言時に

- `int x;`
- `int *p;`
- `int **pp;`

を読んで `LVar->ty` に保存するようにした。

ここで `while (consume("*")) ty = ptr_to(ty);` を使うと、多重ポインタが自然に作れる。

## 10. `add_type()` による型付け

### 10.1 役割

`add_type()` は意味解析に近い。

- parser が AST の形を作る
- `add_type()` が AST に型を付ける

という分離になっている。

### 10.2 再帰の範囲

最初は `lhs` と `rhs` しか見ておらず、`ND_BLOCK` 配下に型が付かなくてセグフォした。

最終的には以下の子ポインタに再帰する必要があると理解した。

- `lhs`
- `rhs`
- `cond`
- `then`
- `els`
- `init`
- `inc`
- `body`
- `next`
- `args`

### 10.3 基本的な型付け

主要ノードは以下のように型を決める。

- `ND_NUM` -> `int`
- `ND_LVAR` -> `node->var->ty`
- `ND_ASSIGN` -> 左辺型
- `ND_ADDR` -> `ptr_to(lhs->ty)`
- `ND_DEREF` -> `lhs->ty->ptr_to`

`ND_DEREF` では `lhs->ty->kind == TY_PTR` か確認している。

## 11. `&` と `*`

### 11.1 構文

`unary()` で

- `&expr`
- `*expr`

を `ND_ADDR`, `ND_DEREF` にしている。

### 11.2 左辺値としての `*p`

`*p = 3` を通すには、`gen_lval()` が `ND_DEREF` を扱える必要がある。

ここでの本質は、

- `gen_lval(ND_LVAR)` は変数スロットのアドレスを積む
- `gen_lval(ND_DEREF)` は `gen(node->lhs)` して、その結果アドレスをそのまま左辺値に使う

ということ。

`ND_DEREF` の左辺値処理で余計な dereference は不要。

## 12. ポインタ引数

`int set5(int *p){ *p = 5; return 0; }` と  
`set5(&a);` が通った。

このとき見えてきたのは、

- 呼び出し側は `&a` を作って `rdi` へ
- 定義側は `rdi` を `[rbp-offset]` へ保存
- `*p = 5` はそのアドレス先へ書き込む

という一連の流れ。

ポインタを関数引数に渡して書き換える実装が通ったことで、ABI と左辺値処理が正しく繋がっていることが確認できた。

## 13. ポインタ加算 / 減算

### 13.1 型による意味分岐

`ND_ADD` / `ND_SUB` は AST 上は普通の `+` / `-` だが、意味は型によって変わる。

- `int + int`
- `ptr + int`
- `int + ptr`
- `ptr - int`

を区別する必要がある。

### 13.2 `add_type()` 側

`ND_ADD` / `ND_SUB` では左右の型を見て、式全体の型を決める。

今の簡易仕様では:

- `int + int` -> `int`
- `ptr + int` -> ptr 型
- `int + ptr` -> ptr 型
- `ptr - int` -> ptr 型
- それ以外 -> エラー

### 13.3 codegen 側

codegen では、ポインタ側ではなく**整数側をスケーリング**する。

たとえば

- `int *p; p + 2`

なら

- `2 * sizeof(int)` してから加算

になる。

`imul reg, imm` を使ってその場で倍化している。

### 13.4 `size_of(Type *ty)`

この段階では簡易版として

- `TY_INT` -> 4
- `TY_PTR` -> 8

を返す関数にした。

ここで重要なのは「ポインタ自身のサイズ」ではなく「指している先の型のサイズ」を使うこと。

例:

- `int *p; p + 1` -> 4バイト進む
- `int **pp; pp + 1` -> 8バイト進む

## 14. `sizeof`

`sizeof` は実行時評価せず、コンパイル時に数値ノードへ潰している。

実装の流れ:

1. `unary()` で `TK_SIZEOF` を読む
2. 対象の `unary()` をパース
3. そのノードに `add_type()`
4. `new_node_num(size_of(node->ty))` を返す

つまり `sizeof` は AST 上の専用ノードにせず、その場で定数化している。

## 15. アドレスとメモリ操作

2周目でかなり腹落ちした点。

- `mov [rax], rdi`
  - `rax` の値をアドレスとして見て、その番地に `rdi` を書く
- `mov rax, [rax]`
  - `rax` の値をアドレスとして見て、その番地の中身を `rax` に読む

`rax` と `[rax]` はまったく別物で、

- `rax` = レジスタの中身
- `[rax]` = その中身をアドレスと見た先のメモリ

という理解が、ポインタ・左辺値・二重ポインタの理解に直結した。

## 16. 実際に通したテスト

### 16.1 ポインタ参照と代入

```c
int main(){ int x; int *y; y=&x; *y=3; return x; }
```

期待値 3。

### 16.2 関数引数としてのポインタ

```c
int set5(int *p){ *p = 5; return 0;} int main(){ int a; a = 1; set5(&a); return a;}
```

期待値 5。

### 16.3 ポインタ加算

`ext.c` の `alloc4` を使って以下を確認した。

```c
int main(){ int *p; alloc4(&p,1,2,4,8); int *q; q=p+2; return *q; }
```

期待値 4。

### 16.4 加算後に書き換えて別ポインタから読む

```c
int main(){ int *p; alloc4(&p,1,2,4,8); int *q; q=p+2; *q=99; int *r; r=p+2; return *r; }
```

期待値 99。

これで

- 同じ場所を指せている
- 書き換えが反映されている
- ポインタ加算のスケーリングも正しい

ことが確認できた。

## 17. 詰まりやすかった点

### 17.1 `add_type()` の再帰不足

`lhs` / `rhs` しか見ていなかったため、block 配下や文列に型が付かず `gen()` でセグフォした。

### 17.2 `expect_kw()` のトークン進行漏れ

`TK_INT` 導入時に、`expect_kw()` の書き方をミスって token が進まず、`expect_ident()` が直後で落ちた。

### 17.3 `ND_DEREF` の扱い

- 単項演算なのに二項演算の generic codegen 経路へ入っていた
- 左辺値としての `*p` で「アドレスを残すだけ」という発想が最初曖昧だった

### 17.4 `locals` と `params`

同じ `LVar` 実体を使いながら、

- `locals` は名前解決用
- `params` は宣言順保持用

でリンクを分離する必要がある点が最初かなり混乱した。

## 18. 現時点で残っている整理ポイント

### 18.1 return の共通ラベル

今は `ND_RETURN` がその場で `ret` まで出すので、関数末尾に到達不能コードが残る。

将来的には

- `.LreturnN`

のような共通ラベルへ飛ばす形にした方がきれい。

### 18.2 `add_type()` のカバー範囲

今は最低限動くところまで来ているが、

- `ND_FUNCALL`
- 制御構文
- 今後の配列・`char`

まで含めると、型付けの一貫性は今後さらに重要になる。

### 18.3 エラーメッセージ層の区別

今は

- tokenize の失敗
- parse の失敗
- meaning/type の失敗

があまり区別されず、「どこで落ちたか」が分かりにくい。

大規模化するなら、

- 字句解析エラー
- 構文解析エラー
- 意味解析エラー

をもっと意識して出し分けたい。

## 19. ここまでで理解が深まったこと

### 19.1 `Type` は「変数型」だけではなく「式型」

`LVar->ty` だけでは足りず、

- `&a`
- `*p`
- `p+1`

のような式全体の型を `Node->ty` として持つ必要がある。

### 19.2 構文解析と意味解析は別

1周目では曖昧だったが、2周目では

- parser は木を作る
- `add_type()` は意味を付ける

という分離がかなり見えた。

### 19.3 ABI に沿っていれば gcc と噛み合う

自作コンパイラで吐いた `call foo` と、gcc で作った関数が普通にリンクして動いたのは、

- 同じ ABI
- 同じ引数レジスタ
- 同じ return 値レジスタ

に従っていたから。

これはかなり印象的だった。

- 関数定義 / 関数呼び出し
