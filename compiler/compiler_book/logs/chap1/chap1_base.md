# 🧠 セッションまとめ（トップダウン・簡潔）

* 今回やったことは、「Cのポインタ（特にポインタのポインタ）が、アセンブリではどういう“アドレス操作”として実現されているか」を、実際の出力を追いながら理解すること

  * 変数・ポインタという概念は消えて、すべて「どのアドレスを読む／書くか」に落ちる

---

* Cで書いたポインタ構造は、メモリ上では単なる「アドレスの連鎖」になる

```c
int x = 10;
int *p = &x;
int **pp = &p;
return **pp;
```

```text
pp → p → x → 10
```

---

* それぞれの変数はスタック上の「rbpからのオフセット」として配置される

  * 変数名はアセンブリでは存在せず、位置だけで管理される

```text
rbp-28 → x
rbp-24 → p
rbp-16 → pp
```

---

* アセンブリでは「アドレスを作る」と「中身を読む」が明確に分かれている

```asm
lea rax, -28[rbp]          ; &x
mov QWORD PTR -24[rbp], rax ; p = &x

lea rax, -24[rbp]          ; &p
mov QWORD PTR -16[rbp], rax ; pp = &p
```

---

* ポインタの参照（デリファレンス）は `[]` で表現される

```asm
mov rax, QWORD PTR -16[rbp] ; pp
mov rax, QWORD PTR [rax]    ; *pp → p
mov eax, DWORD PTR [rax]    ; **pp → x
```

---

* `PTR` はアクセスするサイズを指定するために必要

  * ポインタ自体は 8 byte（QWORD）
  * `*p` のサイズは型に依存（今回は int → DWORD）

---

* `[]` は「そのアドレスの中身を読む」という意味で、Cの `*` と対応している

```asm
mov rax, [rax]  ; *p
```

---

* `-24[rbp]` のような表記は変数ではなく単なるアドレス計算

  * コンパイラだけが「これは p」と知っている
  * アセンブラやCPUは「rbp-24という場所」としか見ていない

---

* コンパイラはコードを機械的に変換するだけで、意味を“理解”しているわけではない

  * 構文に従ってアドレス操作へ変換している

---

* 最適化では「結果が確定しているものは事前に計算する」

```c
int x = 2 + 3;
```

```asm
mov x, 5
```

---

* CPUはさらに実行時に最適化（分岐予測など）を行う

  * 条件分岐の結果を予測して先に実行する

---

* 今回の核心

```text
Cの抽象（変数・ポインタ）
↓
アセンブリ（アドレス + サイズ + 参照）
↓
実体はすべて「メモリ操作」
```

👉 ポインタのポインタも「アドレスを2回たどる」だけであり、特別なものではない
