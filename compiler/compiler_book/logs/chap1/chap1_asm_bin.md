# 低レベル実験まとめ（今回やったこと・得た理解）

* プログラムの実行は
  「コンパイル → アセンブル → リンク → ロード → 実行」という流れで行われるが、今回はその中でも**リンクとランタイムを削った最小構成**を実験した

* アセンブリを直接バイナリにすることで、余計なものがない状態を確認した

  ```sh
  gcc -nostdlib -static my_test_min.s -o a.out
  ```

  * `-nostdlib` により libc を使わない
  * `-static` により動的リンクも排除
  * 結果として「ほぼ自分のコードだけの実行ファイル」になる

* 最小プログラムとして syscall を直接叩くコードを書いた

  ```asm
  .global _start
  _start:
      mov $60, %rax
      xor %rdi, %rdi
      syscall
  ```

  * `syscall` がユーザプログラムとOSの境界
  * printfのような関数は使っていない

* 出力（write）も関数ではなく syscall で行われることを理解した

  ```asm
  mov $1, %rax      # write
  mov $1, %rdi      # stdout
  mov $msg, %rsi    # バッファ
  mov $len, %rdx    # 長さ
  syscall
  ```

  * 出力は「関数呼び出し」ではなく「カーネルへの依頼」
  * データは「ポインタ + サイズ」で扱う

* バイナリ（機械語）はただのバイト列であり、直接書き換えられることを確認した

  * VSCodeのhexエディタで数値を書き換え
  * 即値（immediate）を変えるだけで挙動が変わる
  * CPUがそのバイト列を命令として解釈しているだけ

* 通常のCコードをコンパイルしたときに「知らないコードが大量に入る」ことを観察した

  ```sh
  gcc main.c -o a.out
  objdump -d a.out
  ```

  * 自分が書いたコード以外が大量に含まれる

* その正体は以下であると理解した

  * `_start`（本当のエントリポイント）
  * libc（printfなどの実装）
  * 初期化・終了処理
  * PLT / GOT（動的リンク用の仕組み）

* `main` は実際の入口ではなく、`_start` から呼ばれていることを理解した

  ```text
  _start → 初期化 → main → exit
  ```

  * プログラムの開始点は自分のコードではない

* printf のような関数はコンパイル時には未解決であり、リンク時に結合されることを確認した

  ```sh
  gcc -c main.c
  nm main.o
  ```

  * `U printf`（未定義シンボル）になる
  * リンクで libc と結合されて初めて使える

* 動的リンクでは関数呼び出しが間接的になることを理解した

  ```text
  call printf
    ↓
  PLT
    ↓
  GOT
    ↓
  libc
  ```

  * 実行時にアドレスが決まる

* 「余計なコード」は不要ではなく、実行に必要なランタイムであると理解した

  ```text
  実行ファイル =
    自分のコード
    + 起動コード（_start）
    + ライブラリ
    + リンク用テーブル
  ```

* それらを削ることで、プログラムの本質構造が見えることを確認した

  * `-nostdlib` でライブラリ削除
  * `_start` を自分で書くことで完全制御

* 今回の実験で得た最も重要な理解

  * 機械語は単なるバイト列である
  * 意味はCPUが解釈している
  * 高級言語の便利さは「大量の裏側のコード」によって実現されている
  * プログラムは単体ではなく「実行環境込み」で成立している
