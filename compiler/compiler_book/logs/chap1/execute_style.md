# コンパイル・実行モデルまとめ

## 1. 全体フロー

```text
ソースコード
↓（コンパイル）
オブジェクトファイル（.o）
↓（リンク）
実行ファイル（ELFなど）
↓（ロード）
メモリ上で実行
```

---

## 2. 各段階の役割

### 2.1 コンパイル

* ソースコード → 中間表現 → アセンブリ
* 外部関数は未解決のまま

例：

```c
printf("hi\n");
```

↓

```text
printf（未解決シンボル）
```

---

### 2.2 アセンブル

* アセンブリ → 機械語（.o）
* まだ単体では実行不可

---

### 2.3 リンク（重要）

* 複数のオブジェクトを結合
* 未解決シンボルを解決
* 標準ライブラリがここで結合される

#### 静的リンク

```text
main.o + libc.a → 完全な実行ファイル
```

#### 動的リンク

```text
main.o → 「libcを使う」という情報だけ持つ
```

---

### 2.4 ロード（OSの仕事）

* 実行ファイルをメモリに配置
* 必要な共有ライブラリを読み込む
* アドレスを最終決定
* エントリポイントから実行

---

## 3. 実行ファイルに含まれる「余計なもの」の正体

機械語に含まれていた追加要素：

* `_start`（実際のエントリポイント）
* libc 初期化コード
* 終了処理
* PLT（Procedure Linkage Table）
* GOT（Global Offset Table）

---

### `_start` の役割

```text
_start:
  初期化処理
  main呼び出し
  exit処理
```

重要：

* `main` は直接実行されない
* `_start` が最初に呼ばれる

---

## 4. 標準ライブラリ（printf など）の扱い

例：

```c
printf("hello\n");
```

### コンパイル時

```text
printf は未解決
```

---

### リンク時

```text
printf の実体を libc から取得
```

---

### 実行時（動的リンクの場合）

```text
printf のアドレスを決定
```

---

## 5. 再帰 vs ループ（低レベル視点）

### forループ

```c
for (int i = 0; i < n; i++) { ... }
```

↓

```text
cmp
jmp
inc
jmp
```

👉 ジャンプベース

---

### 再帰

```c
f(i + 1);
```

↓

```text
call
ret
```

👉 スタックを使う

---

### 末尾再帰

最適化あり：

```text
call → jmp に変換
```

---

## 6. 言語ごとの実行モデル

### C

```text
ソース → アセンブリ → ネイティブ
```

* ほぼ直接実行

---

### Java

```text
ソース → バイトコード → JVM → JIT → ネイティブ
```

---

### Python

```text
ソース → AST → バイトコード → インタプリタ実行
```

* 型は実行時に決定

---

### JavaScript

```text
AST → バイトコード → JIT → ネイティブ
```

---

## 7. 中間表現の違い

### JVMバイトコード

* スタックマシン

```text
push
push
add
```

---

### LLVM IR

* レジスタマシン（SSA）

```llvm
%1 = add i32 1, 2
```

---

### SSA（Static Single Assignment）

```text
x1 = 1
x2 = x1 + 2
```

* 変数は一度しか代入されない
* 最適化しやすい

---

## 8. 関数型言語（Haskell）の違い

### 特徴

* 遅延評価
* 式ベース
* クロージャ・thunk

---

### コンパイルフロー

```text
AST
↓
Core（関数的IR）
↓
STG（遅延評価モデル）
↓
C--（低レベル）
↓
アセンブリ
```

---

### 本質的な違い

|       | C    | Haskell  |
| ----- | ---- | -------- |
| 評価    | 即時   | 遅延       |
| 値     | 実体   | 計算       |
| ランタイム | ほぼ不要 | 必要（GCなど） |

---

## 9. 動的型言語の本質

例：

```python
a + b
```

---

### AST

```text
BinOp(a, +, b)
```

---

### 実行時

```text
a.__add__(b)
```

👉 型は実行時に決定

---

## 10. CUDA の実行モデル

```text
CUDAコード
↓
PTX（中間表現）
↓
GPUネイティブコード
↓
GPU実行
```

---

### PTXの特徴

* GPU用中間言語
* アセンブリ寄り
* 仮想ISA

---

### JIT的要素

```text
PTX → 実行時にGPUコードへ変換
```

---

## 11. 重要な統一視点

すべての言語は：

```text
構文（AST）
↓
意味付け（型・評価）
↓
中間表現
↓
機械語
```

---

### 違いはここ

* いつ意味を決めるか
* どの中間表現を使うか
* 実行時にどれだけ仕事を残すか

---

## 12. 核心まとめ

* 機械語は「自分のコードだけ」ではない
* 実際は：

```text
自分のコード
+ ランタイム
+ ライブラリ
+ 起動コード
```

---

* 標準ライブラリは主にリンク時に結合される
* 動的リンクの場合は実行時にも関与
* ローダはメモリ配置と最終解決を行う

---

## 13. 実験で確認できること

### 未解決シンボル確認

```sh
gcc -c main.c
nm main.o
```

→ `U printf`

---

### 実行ファイル確認

```sh
objdump -d a.out
```

---

### ELF構造確認

```sh
readelf -a a.out
```
